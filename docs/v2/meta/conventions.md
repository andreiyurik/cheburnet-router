# 🧭 Как устроен этот vault

Документация спроектирована по принципам **Andrej Karpathy**: писать так, чтобы было удобно
и человеку, и LLM; объяснять **от первых принципов**; никаких чёрных ящиков; всё
copy-paste-able.

## Принципы (по ним пишем каждую заметку)

1. **Атомарность.** Одна заметка = одна концепция. Если тема распухает — дроби и связывай
   `[[wikilink]]`'ами. Маленькие связанные заметки > одна простыня.
2. **От первых принципов.** Объясняем *почему* и *как на самом деле*, а не только *что нажать*.
   Цель — чтобы читатель мог донастроить сам.
3. **Без чёрных ящиков.** Если что-то «просто работает» — раскрываем механику до примитива
   ядра (nftables, ip rule, dnsmasq).
4. **Структура What → How → Why.** Сначала что это, потом как работает, потом почему так
   решили (или ссылка на ADR).
5. **Copy-paste-able.** Команды — целыми блоками, готовыми к запуску. Никаких интерактивных
   виджетов и скриншотов вместо текста (LLM их не прочитает).
6. **Плоский markdown.** Mermaid для схем (Obsidian и GitHub рендерят), таблицы, callout'ы.
   Ничего, что ломается при копировании в текст.

## Анатомия заметки

```markdown
---
title: Человекочитаемое название
tags: [concept | architecture | decision | reference | meta, тема]
aliases: [короткое-имя, синоним]   # для wikilink'ов и поиска
updated: YYYY-MM-DD
related: ["[[другая-заметка]]"]      # опционально
---

# Заголовок

> [!tip] TL;DR
> Одно-два предложения сути — чтобы и человек, и ИИ схватили мысль сразу.

## Что это
## Как работает   (с mermaid/командами)
## Почему так      (или → [[ADR]])
## Дальше          (ссылки [[...]])
```

## Теги (таксономия)

- `#concept` — сетевой примитив / «как работает»
- `#architecture` — как примитивы собраны в систему
- `#decision` — ADR
- `#reference` — справка, глоссарий
- `#meta` — про сам vault
- Тематические: `#data-plane #dns #routing #security #testing`

## Wikilinks

Связываем **щедро**: ссылка `[[имя]]` на ещё не написанную заметку — это нормально, она
помечает «стоит написать». Используем basename файла без пути: `[[policy-routing]]`,
а не `[[concepts/policy-routing]]`.

# 🤖 Для ИИ

Vault сделан так, чтобы быть хорошим **контекстом для LLM**:

- **Чистый markdown** — копируется в промпт без потерь.
- **Атомарность** — можно дать модели ровно нужные заметки, не весь корпус.
- **Frontmatter `related` + `[[links]]`** — модель видит связи и может попросить смежное.
- **Self-contained заметки** — каждая объясняет свою концепцию целиком, без «см. выше».

## Как собрать контекст одним куском

Конкатенация всех заметок в один файл для скармливания модели:

```bash
# из корня репо
find docs/v2 -name '*.md' | sort | while read -r f; do
  printf '\n\n===== FILE: %s =====\n\n' "$f"; cat "$f"
done > /tmp/cheburnet-v2-context.md
```

Получится один markdown со всей базой знаний и явными разделителями файлов — удобно дать
ИИ как единый контекст.

> [!note] Принцип «docs as context»
> Мы относимся к документации как к **контекст-инженерии**: заметки пишутся так, чтобы
> модель могла на них опереться и не галлюцинировать. Поэтому — точные команды, явные
> инварианты, ссылки на [[glossary]] вместо неявных предположений.
