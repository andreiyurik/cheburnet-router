# 🔧 Troubleshooting — куда смотреть

> [!tip] TL;DR
> 90% жалоб «нет интернета» решаются тремя командами: статус [[amneziawg|туннеля]], статус
> режима, последние логи. Начинай с них.

## Первые три команды

```bash
vpn-mode status                      # режим (HOME/TRAVEL) + общее состояние
awg show awg0 | grep handshake       # туннель живой? «N seconds ago» < 2 мин = ок
logread | tail -200                  # что случилось недавно
```

## По симптому

### «Нет интернета вообще»
1. `awg show awg0` — есть свежий handshake? Нет → проблема с [[amneziawg|туннелем]]/конфигом.
2. [[kill-switch]] работает как задумано: туннель упал → непрямой трафик дропается (не утечка).
   Это **не баг**, а защита. Чини туннель.

### «Сайт из direct-списка идёт через VPN, а должен напрямую»
Проверь, попал ли его адрес в [[dnsmasq-nftset|direct]]:
```bash
nft list set inet fw4 direct         # есть ли там нужный IP
nslookup example.com                  # резолвится через наш dnsmasq?
```
Если домена нет в твоём direct-списке (или в импортированном community-списке) — его адрес
туда не попадёт. См. [[dnsmasq-nftset]].

### «Реклама лезет»
```bash
# проверь, что adblock активен и список загружен
logread -e adblock
```
См. [[adblock]].

### «Клиент не слушается split-routing»
Возможно, у устройства **свой DoH** (Chrome, смарт-ТВ) — оно не резолвит через нас.
Это [[dns-and-routing#Зависимость от того, что клиент использует НАШ DNS|известный размен]]:
такой клиент уходит в туннель (fail-safe), прямого пути для доменов из списка у него не будет.

## Проверка плоскости данных

```bash
ip rule show                         # видны ли наши policy-routing правила
ip route show table 100              # таблица «напрямую через WAN»
nft list ruleset | less              # все правила фаервола и kill-switch
```

Поскольку весь [[data-plane]] — на нативных примитивах, всё **видно и проверяемо** командами.
Это прямое следствие «без чёрных ящиков» ([[0001-why-not-singbox]]).

## Логи установки

