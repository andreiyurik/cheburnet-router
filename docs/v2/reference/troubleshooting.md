# 🔧 Troubleshooting — куда смотреть

> [!tip] TL;DR
> 90% жалоб «нет интернета» решаются тремя командами: статус [[amneziawg|туннеля]], статус
> режима, последние логи. Начинай с них.

## Первые три команды

```bash
vpn-mode status                      # режим (HOME/TRAVEL) + общее состояние
awg show awg0 | grep handshake       # туннель живой? «N seconds ago» < 2 мин = ок
logread | tail -200                  # что случилось недавно
```

## По симптому

### «Нет интернета вообще»
1. `awg show awg0` — есть свежий handshake? Нет → проблема с [[amneziawg|туннелем]]/конфигом.
2. [[kill-switch]] работает как задумано: туннель упал → непрямой трафик дропается (не утечка).
   Это **не баг**, а защита. Чини туннель.

### «Сайт из direct-списка идёт через VPN, а должен напрямую»
Проверь, попал ли его адрес в [[dnsmasq-nftset|direct]]:
```bash
nft list set inet fw4 direct         # есть ли там нужный IP
nslookup example.com                  # резолвится через наш dnsmasq?
```
Если домена нет в твоём direct-списке (или в импортированном community-списке) — его адрес
туда не попадёт. См. [[dnsmasq-nftset]].

### «Реклама лезет»
Блокировку даёт **выбранный DNS-провайдер**, а не локальный список — проверь, что выбран
фильтрующий резолвер и что запросы реально идут через него:
```bash
uci get https-dns-proxy.cheburnet_doh.resolver_url   # какой резолвер выбран
nslookup ads.example.com                              # фильтрующий вернёт пустой/0.0.0.0
```
Не фильтрует → смени провайдера в панели (`ads`/`family`). Учти: реклама с общего CDN-домена
DNS-блокировкой не режется. См. [[adblock]], [[0005-dns-filtering-not-local-adblock]].

### «Клиент не слушается split-routing»
Возможно, у устройства **свой DoH** (Chrome, смарт-ТВ) — оно не резолвит через нас.
Это [[dns-and-routing#Зависимость от того, что клиент использует НАШ DNS|известный размен]]:
такой клиент уходит в туннель (fail-safe), прямого пути для доменов из списка у него не будет.

### Мастер вместо шага «Проверка роутера» показывает экран смены LAN-адреса

Это не баг — конфликт подсетей LAN/WAN, детект **до** запуска обычного мастера. Возникает,
когда cheburnet-router висит за другим роутером (каскад), и обе стороны используют одну и ту
же подсеть `192.168.1.0/24` — тогда policy routing и kill-switch не могут отличить «локальную»
сеть от «внешней» через LAN-интерфейс.

Как это работает:
1. При заходе на мастер (до preflight) фронтенд зовёт ubus-метод `check_lan_conflict` (anon,
   без токена) — движок сверяет `lan_cidr`/`wan_cidr` из `network.interface.*.status` через
   `cidr_overlap()` (сравнение сетевой части по меньшему из двух префиксов).
2. При конфликте ответ содержит `suggest_ip` — первый свободный кандидат из
   `192.168.{2,3,4,8,9,10,11}.0/24` (перебираются по порядку, каждый проверяется той же
   `cidr_overlap` против WAN), никогда не совпадающий с диапазоном WAN.
3. Экран `LanConflict.svelte` показывает оба CIDR и предложенный адрес; можно **пропустить**
   (конфликт останется — предупреждение будет разве что в основном preflight, который тоже
   блокирует на `lan_wan`) или **сменить LAN** — это зовёт `apply_lan_ip` (write, требует
   install-токен), который валидирует адрес (`valid_lan_ip`: строго `192.168.X.Y`, хост
   `1..254`), правит `network.lan.ipaddr` (сохраняя маску) и через 3 секунды перезапускает
   `network` — роутер станет доступен по новому адресу, старая сессия браузера оборвётся.

Если застрял на этом экране без явной причины — сверь `192.168.1.x` со своим апстрим-роутером
(тот, что раздаёт cheburnet-router интернет по WAN).

## Проверка плоскости данных

```bash
ip rule show                         # видны ли наши policy-routing правила
ip route show table 100              # таблица «напрямую через WAN»
nft list ruleset | less              # все правила фаервола и kill-switch
```

Поскольку весь [[data-plane]] — на нативных примитивах, всё **видно и проверяемо** командами.
Это прямое следствие «без чёрных ящиков» ([[0001-why-not-singbox]]).

## Логи установки

```bash
cat /tmp/cheburnet/install.log       # лог последней установки
cat /tmp/cheburnet/done              # маркер завершения
```
