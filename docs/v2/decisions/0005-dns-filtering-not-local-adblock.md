# ADR 0005 — Фильтрация = выбор DNS-провайдера

> [!info] Статус: принято · реализовано (Light-тир) · 2026-06-12
> Заменяет локальный adblock (adblock-lean) и семейный фильтр (NSFW-список + SafeSearch-cname)
> единым механизмом: пользователь выбирает фильтрующий DoH-резолвер.

## Контекст

В v1/раннем v2 блокировка рекламы и взрослого контента делалась **локально на роутере**:
adblock-lean грузил в dnsmasq список из ~250k доменов, семейный режим добавлял NSFW-URL +
cname-перенаправления на SafeSearch. У этого подхода для нашей задачи (надёжный лёгкий роутер
для семьи, в т.ч. на слабом старом железе) три проблемы:

1. **adblock-lean не в apk-feed** — ставится своим скриптом с GitHub → `apk add cheburnet`
   падал; «vendored-fallback долг» (ручное устаревание), о котором предупреждает CLAUDE.md.
2. **Вес** — большой блок-лист в dnsmasq = 15–40 МБ RAM, бьёт по **самому слабому тиру** (тому,
   ради лёгкости которого всё затевалось), угрожая «годам без обслуживания» (OOM).
3. **Не миссия** — блокировка рекламы вторична к VPN/обходу; не должна блокировать установку
   или рисковать стабильностью ядра.

## Решение

**Блокировка делается на стороне DNS — выбором фильтрующего DoH-резолвера.** Поскольку движок и
так гонит весь DNS через [[encrypted-dns|https-dns-proxy]], «фильтрация» = какой резолвер выбран.

- **Каталог провайдеров** ([providers.uc](../../../engine/steps/doh/providers.uc)) — дружелюбные
  описания + DoH-URL. Пользователь выбирает по короткому описанию (Setup и панель Status).
- **Категории = классы фильтрации:** `plain` (без), `ads` (реклама+трекеры), `family` (+18+ и
  безопасный поиск). «Семейный режим» = выбрать семейного провайдера (AdGuard Семейный /
  CleanBrowsing Семейный) — он сам форсит SafeSearch на стороне DNS.
- **Дефолт — AdGuard** (реклама+трекеры): полезно и ничего не ломает.
- ubus `set_dns_provider(id)` + `dns_provider` в install (enum из каталога — граница доверия).

Удалены: локальный шаг adblock (adblock-lean) и шаг family (NSFW+cname). Их заменяет выбор провайдера.

## Почему это лучше для нашей задачи

| Критерий | Локальный adblock | Выбор DNS-провайдера |
|---|---|---|
| Вес на роутере | список в RAM (15–40 МБ) | **~ноль** (фильтрует провайдер) |
| Слабый старый роутер | сужает | **тянет** |
| Обслуживание | обновлять списки | **ноль** (провайдер обновляет) |
| Установка | adblock-lean ломал `apk add` | чистая (нет лишних DEPENDS) |
| Простота для пользователя | тиры/тумблеры | **один список с описаниями** |

## Надёжность (важная тонкость)

DNS не должен **умереть** (потеря интернета хуже рекламы) и семейный фильтр не должен молча
**расфильтроваться**. Поэтому:

- Каждый провайдер несёт **два anycast-эндпоинта** (bootstrap) — redundancy ВНУТРИ провайдера,
  а не кросс-классовый fallback на не-фильтрующий резолвер.
- Одна секция `cheburnet_doh` (фиксированное имя) → смена провайдера = чистая идемпотентная
  замена (delete-before-set), без накопления секций.
- Неизвестный id → дефолт каталога (fail-safe: лучше рабочий DNS, чем пустой).

## Честные размены

> [!warning] Что принимаем
> - **Доверие резолверу:** он видит запросы. Но DoH и так шлёт DNS какому-то резолверу —
>   меняем на тот, что заодно фильтрует. Не новая потеря; дефолт — no-log ad-фильтр, выбор
>   провайдера остаётся за пользователем.
> - **Меньше тонкого контроля:** whitelisting у бабушки сложнее, чем локально. Но «просто
>   работает» обычно лучше агрессивного локального списка.
> - **CDN-реклама** на общем домене не режется — общая беда любой DNS-блокировки (и у adblock-lean
>   так же).
> - Зависимость от доступности провайдера; anycast-гиганты почти не падают целиком.
