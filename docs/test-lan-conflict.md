# 🧪 Тестирование LAN/WAN-conflict-фичи

Шпаргалка по ручной проверке детекта и автофикса конфликта подсетей LAN/WAN
(каскад «главный роутер → cheburnet-router», когда оба используют 192.168.1.x).

**Аудитория:** мейнтейнер + контрибьюторы, которые трогают что-то в
`lib/net-detect.sh`, `lib/cheburnet-preflight.sh`, `setup.sh`,
`setup/install.sh`, `web/rpcd-cheburnet`, `web/index.html` относящееся к
LAN-conflict-логике.

**Когда применять:** после любых изменений в перечисленных файлах, перед
merge'ом в `master`.

---

## Что вообще тестируем

Фича двухслойная:

1. **Pre-check** — детект ДО запуска install. Делается в двух местах:
   - `setup.sh` (CLI-путь) — `_conflict_info=$(ssh ...)` после Шага 1.
   - `web/index.html` (Web-путь) — `checkLanConflict()` в `init()`.
   Если конфликт есть, мастер предлагает сменить LAN-IP, юзер
   переподключается, дальше всё идёт нормально.

2. **Safety net** — `cheburnet_preflight_lan_conflict` в `setup/install.sh`.
   Срабатывает если WAN получил аренду уже после pre-check'а. Не чинит сам —
   печатает баннер с инструкцией ручного фикса.

**Источник истины** для логики детекта — `net_detect_lan_conflict` в
`lib/net-detect.sh`. Эту функцию вызывают все три слоя.

---

## Уровень автоматики — что НЕ нужно делать руками

Эти уровни прогоняются автоматически (CI или `make`), руками их повторять
не нужно — только смотреть, что прошли:

| Команда | Время | Покрывает |
|---|---|---|
| `make lint` | <1 с | shellcheck, JSON, manifest |
| `make test` | ~3 с | 16 bats-тестов на детектор + RPC-методы (`tests/integration/test_lan_conflict.bats`) |
| `make qemu-install` | ~10 мин | sub-test на реальном busybox-ash + jsonfilter (4 сценария в `tests/qemu/install.sh`) |

Если эти три зелёные — логика shell-функций и RPC проверена в максимуме того,
что покрывается автоматами. **Дальше — то, что автоматика не достанет:**
веб-UI визуально, end-to-end на железе, поведение DHCP клиента после
смены IP.

### Сценарии с реальным железом (одна железка, без каскада)

**Преконды для всех сценариев:**
- Один чистый OpenWrt 25.12+ роутер (Beryl AX / Cudy / etc.).
- Ноутбук подключён LAN-кабелем к роутеру.
- Корень репо cheburnet-router на ноуте.
- Если стоит OPENWRT_BLOCKED / install — сделать factory reset (см. `tests/hardware/README.md` чек-лист).

#### Сценарий A — CLI-путь (`setup.sh`)

```bash
# 1. Создаём искусственный конфликт через ssh (без второго роутера).
# Берём 192.168.1.50 как «WAN от главного роутера», LAN остаётся 192.168.1.1.
ssh root@192.168.1.1 '
    uci set network.wan.proto=static
    uci set network.wan.ipaddr=192.168.1.50
    uci set network.wan.netmask=255.255.255.0
    uci set network.wan.gateway=192.168.1.1
    uci commit network
    /etc/init.d/network restart
'
sleep 8   # network restart на DSA-роутерах долгий

# 2. Запускаем мастер.
cd cheburnet-router
./setup.sh
```

**Что должно произойти:**
1. После «Адрес роутера» (Enter на default) скрипт печатает
   ```
   Найден конфликт подсетей роутера:
       • WAN роутера: 192.168.1.50   (получен от главного роутера)
       • LAN роутера: 192.168.1.1    (по умолчанию OpenWrt)
       ...
       1. Поменяю LAN-IP роутера: 192.168.1.1 → 192.168.2.1
   ```
2. После Enter — реально сменил IP (`uci get network.lan.ipaddr` на роутере
   = `192.168.2.1`).
3. Просит переподключить кабель. **Отсоединяешь и присоединяешь** кабель
   к ноуту (или Wi-Fi, если по Wi-Fi). Жмёшь Enter.
4. Скрипт ловит роутер на 192.168.2.1 в течение ≤60 сек.
5. Дальнейший мастер идёт нормально (запрашивает VPN-конфиг, пароль и т.д.).
6. **В preflight install.sh новых баннеров о конфликте быть не должно.**

**Edge case A1 — не переподключать кабель.** Жмёшь Enter не вынимая кабель.
Ноут остаётся в 192.168.1.x, не видит 192.168.2.1. Скрипт должен:
- Подождать 60 сек (retry-loop).
- Умереть с понятным сообщением и инструкцией про release/renew DHCP под
  Linux/macOS/Windows.
- **НЕ** зависнуть навсегда.

**Edge case A2 — нажать Ctrl+C на этапе prompt'а смены.** Не должно сломать
роутер: LAN-IP остаётся 192.168.1.1, мастер выходит чисто. (До нажатия
Enter мы ничего не меняли.)

#### Сценарий B — Web-путь

```bash
# 1. Свежий роутер + bootstrap.
ssh root@192.168.1.1 'wget -qO- https://raw.githubusercontent.com/andreiyurik/cheburnet-router/master/install.sh | sh'
# Запоминаем install-token из вывода (32 hex).

# 2. Создаём конфликт через ssh (как в сценарии A, шаг 1).
ssh root@192.168.1.1 'uci set network.wan.proto=static; uci set network.wan.ipaddr=192.168.1.50; uci set network.wan.netmask=255.255.255.0; uci commit network; /etc/init.d/network restart'
sleep 8

# 3. Открыть в браузере:
# http://192.168.1.1/cheburnet/?token=ВАШ_ТОКЕН
```

**Что должно произойти:**
1. Вместо обычного welcome-экрана → экран «⚠ Перед установкой нужно
   сменить адрес роутера».
2. Видим WAN_IP, LAN_IP, suggest_ip с реальными значениями.
3. Поле токена скрыто (токен пришёл из URL).
4. Кликаем «Сменить адрес роутера» → экран «Меняю адрес…» с инструкциями.
5. Через ~10 сек переподключаемся к Wi-Fi/кабелю.
6. Жмём «Открыть новый адрес» → попадаем на http://192.168.2.1/cheburnet/?token=...
7. На новом адресе — обычный welcome без баннера конфликта.

**Edge case B1 — без `?token=` в URL.** Удалить `?token=...` из URL до
открытия. На экране конфликта должно появиться поле «Install-токен», и
кнопка «Сменить» должна не работать пока 32 hex-символа не введены.

**Edge case B2 — невалидный токен.** Ввести 32 нуля. После клика — должно
вывести ошибку «invalid install token», кнопка снова активна.

**Edge case B3 — браузер на новом адресе сразу.** После клика «Сменить» НЕ
переподключаться к Wi-Fi/кабелю. Клик «Открыть новый адрес» → должна
получиться ошибка «не удалось подключиться» (естественно — ноут ещё на
старом IP). Это нормальное поведение — наша помощь в этом случае была
адекватной.

#### Сценарий C — happy path БЕЗ конфликта (anti-regression)

**Самое важное проверить!** Что мы не сломали обычный сценарий.

```bash
# 1. Свежий роутер. WAN подключен к источнику с IP в ДРУГОЙ /24 (например,
#    телефон в режиме модема — обычно 192.168.43.x или 192.168.137.x).
# 2. Запускаем мастер:
cd cheburnet-router
./setup.sh
```

**Что должно произойти:**
- После «Адрес роутера» — должна напечататься одна строка
  `→ Проверяю, не конфликтуют ли подсети LAN и WAN роутера...` и сразу
  переход к Шагу 2 (VPN-конфиг).
- **Никаких баннеров о конфликте.**

То же самое в web: открываем http://192.168.1.1/cheburnet/?token=... →
сразу попадаем на conf-экран без промежуточных страниц.

#### Сценарий D — safety net в preflight

Узкий edge case: WAN получил аренду уже **во время** install.sh.

```bash
# 1. Свежий роутер. WAN-кабель ОТКЛЮЧЁН (не воткнут вообще).
# 2. Запускаем setup.sh — pre-check молча проходит (нет WAN).
# 3. На Шаге 5 (Подтверждение) перед нажатием Enter — НЕ ОТКРЫВАЯ ТЕРМИНАЛ:
#    подключаем WAN-кабель в каскад (главный роутер той же /24, что и LAN).
# 4. Жмём Enter, начинается установка. К моменту дойти до preflight
#    в install.sh, WAN уже получил DHCP-аренду 192.168.1.X.
```

**Что должно произойти:**
- install.sh падает с большим баннером:
  ```
  ════════════════════════════════════════════════════════════
    ❌ РОУТЕР НЕ ПОДХОДИТ ДЛЯ CHEBURNET-ROUTER
    КОНФЛИКТ ПОДСЕТЕЙ LAN И WAN
    ...
    ЧТО ДЕЛАТЬ ВРУЧНУЮ:
      1. На роутере (через SSH):
           uci set network.lan.ipaddr=192.168.2.1
           ...
  ════════════════════════════════════════════════════════════
  ```
- `/tmp/cheburnet/done` содержит `fail-preflight-lan-conflict`.
- **Никаких автоматических смен IP** (мы НЕ хотим рвать сеть посреди
  установки).

Это сценарий — паранойя; в реальности почти не встретится. Но если идёт
релиз — лучше один раз руками пройти, чтобы убедиться, что safety net
действительно срабатывает (а не молча пропускает).

## Сценарий с двумя роутерами (если есть)

Идеальный финальный тест: подключить cheburnet-роутер WAN-портом в LAN-порт
**настоящего** Keenetic/Mi/etc, у которого подсеть 192.168.1.x. Это тот
самый сценарий, ради которого всё писалось.

Прогон обоих путей (CLI и Web) end-to-end без uci-подделок. Если работает
без замечаний — продакшн-готово.

Особенное внимание:
- **Смартфон через Wi-Fi после смены IP:** переподключение к Wi-Fi работает
  естественно или менее технический пользователь застрянет?
- **macOS / Linux / Windows:** время DHCP-renew разное. 60-сек retry-loop
  должен покрыть, но желательно проверить хотя бы один не-Linux клиент.

## Что делать если что-то сломалось

Откатить uci-подделку WAN к DHCP:

```bash
ssh root@<router-ip> '
    uci set network.wan.proto=dhcp
    uci -q delete network.wan.ipaddr
    uci -q delete network.wan.netmask
    uci -q delete network.wan.gateway
    uci commit network
    /etc/init.d/network restart
'
```

Если случайно сменили LAN-IP, а ноут не получает новый адрес —
release/renew DHCP:

| OS | Команда |
|---|---|
| Linux (NetworkManager) | `sudo nmcli con down "Wired connection 1" && sudo nmcli con up "Wired connection 1"` |
| Linux (dhclient) | `sudo dhclient -r && sudo dhclient` |
| macOS | System Settings → Network → ваш интерфейс → Renew DHCP Lease |
| Windows | `ipconfig /release && ipconfig /renew` |

Если вообще ничего не помогает — отсоединить кабель на 10 сек, подсоединить
обратно. Большинство OS воспринимают carrier-flap как сигнал «нужно
запросить новую аренду».

## Что НЕ покрыто этим документом

- **IPv6-only WAN.** Детектор смотрит только на IPv4. Если родительский
  роутер раздаёт только IPv6 (редкий случай в SOHO) — конфликта не будет
  по определению, детектор молча возвращает «нет конфликта».
- **/16 или /22 подсети.** Детектор сравнивает только первые три октета
  (/24). У домашних провайдеров это всегда так, у corporate — может быть
  иначе. Если попадёшь — добавь сценарий в `test_lan_conflict.bats`.
- **Двойной NAT с тройным роутером (Keenetic → Beryl AX → ?).** Не
  тестировали, но логически должно работать так же: каждый уровень видит
  WAN/LAN своего уровня, конфликт детектится попарно.
