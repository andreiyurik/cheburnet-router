# tests/netns — поведение split-routing на живом ядре (без роутера)

`dataplane.sh` отвечает на вопрос «а точно ли после установки трафик разделяется правильно и
**не утекает**?» — детерминированно, за секунды, **без роутера, QEMU и рабочего VPN**.

## Ключевая идея: split — это локальное решение ядра

Разделение трафика (какой интерфейс берёт пакет) и kill-switch — **чисто локальная** механика:
nftset-членство + `fwmark` + `ip rule` + `drop`-правило. Она **не зависит от VPN-крипты**. Значит
туннель можно подменить `dummy`-интерфейсом и проверять само *решение о маршруте*, наблюдая egress
через nft-счётчики. Живость самого туннеля (реальные зашифрованные байты) — отдельная забота
(AWG — можно поднять реального пира; Reality — на железе с VPS, см. ADR 0004).

## Почему форвард-путь, а не output (в отличие от `poc/split-routing-netns.sh`)

Продакшн метит трафик в **PREROUTING**, а **kill-switch живёт в FORWARD-хуке** (см.
`engine/steps/firewall/firewall.uc`). Локально-сгенерированный трафик эти хуки **не проходит** →
kill-switch там непроверяем в принципе. Поэтому строим настоящую топологию
`client → router → {wan0, tunnel}` и гоним **форвард-трафик** клиента.

```
     netns client                 netns router (unshare -rn)
   ┌────────────┐   veth    ┌──────────────────────────────────┐
   │  10.0.0.2  ├──────────►│ 10.0.0.1  ip_forward=1            │
   └────────────┘           │   ├─ nft: mark(prerouting)+ks(fwd)│  ← РЕАЛЬНЫЙ вывод движка
      ping direct/other     │   ├─ ip rule fwmark→table100      │     (tests/netns/emit.uc)
                            │   ├─ dummy wan0 ─┐ счётчик c_wan   │
                            │   └─ dummy tunN ─┘ счётчик c_tun   │
                            └──────────────────────────────────┘
```

Счётчики-наблюдатели висят на forward-хуке с `priority 200` (после kill-switch `filter=0`): `drop`
туда не долетает, поэтому инкремент `c_wan` = трафик **реально** ушёл в WAN (утёк). `dummy`-туннель
роняет пакет на xmit, но форвард-хук egress уже оценил — этого достаточно.

## Что проверяется

- **Split**: direct-адрес → WAN напрямую; остальное → туннель. Для `awg0` **и** `singtun0`.
- **Kill-switch (антиутечка)**: туннель «упал» (netifd снял default) → непрямой трафик **дропается**,
  не течёт в открытый WAN; direct при этом продолжает работать.
- **Travel**: весь трафик в туннель, любой выход в WAN зарублен.
- **Идентичность data-plane** awg vs reality: nft-правила побайтово равны (туннель взаимозаменяем).
- **Мост «домен→IP→set»**: настоящий `dnsmasq` с нашей nftset-строкой резолвит direct-домен →
  IP попадает в `@direct` → маршрут уходит в WAN; непрямой домен в set не попадает и идёт в туннель.
  (Главный шрам v1 — семантика direct-списка.)

Тест гоняет **реальный вывод движка** (`emit.uc` → `build_firewall_plan` / `render_dnsmasq`), а не
переписанную в тесте копию правил — иначе проверяли бы тест, а не продакшн.

## Запуск и изоляция

Всё **rootless** (`unshare -rn` + дочерний netns по PID) — как `poc-split`, без `sudo`. Каждый
сценарий — в **свежем** namespace (ре-exec `__run`), чтобы состояние не протекало между ними.
Каждый probe идёт на новый адрес (`203.0.113.N` / `198.51.100.N`) → `ct state new` гарантирован
без conntrack-flush.

```sh
make test-netns              # локально
NETNS_REQUIRE=1 make test-netns   # CI: нехватка инструментов = ФЕЙЛ, а не тихий скип
```

Подтест **membership** требует `dnsmasq` + `nslookup`/`dig`; без них он честно пропускается
(валидируется в CI — см. `.github/workflows/engine.yml`, job `netns-dataplane`). Роутерный
data-plane на живом OpenWrt/`fw4`/`netifd` дополнительно проверяют QEMU-тесты (`tests/qemu`).
