# engine/steps/rootpass — пароль администратора роутера

Ставит пароль пользователя `root` — учётка администратора OpenWrt (SSH/LuCI). В веб-мастере
это отдельные поля; шаг применяет введённое значение через busybox `passwd`.

## Чистое ядро vs импурный apply

- **`rootpass.uc`** — `validate_password(pw)` → `{ ok, errors }`. **Чистая функция**: правило
  одно — непустая строка не короче 8 символов. Юнит-тесты без роутера ([tests/](tests/)).
- **`apply.uc`** — **router-side, импурно**: читает `{root_password}` JSON'ом со stdin,
  валидирует тем же ядром и зовёт `passwd root` (busybox читает новый пароль **дважды** со
  stdin — приём из v1). Значение пароля **не логируем**. Проверяется в QEMU.

Здесь нет `plan.uc`: у смены пароля нет конфиг-diff'а, который имело бы смысл «спланировать» —
вся «логика» это валидация. Не плодим файл ради симметрии.

## Почему не откатывается

Смена пароля root — runtime-операция над `/etc/shadow`, не uci, и **всегда безопасное
улучшение**. Поэтому шаг не входит в uci-снимок установки и не маскируется под транзакцию:
оркестратор ([install/run.uc](../../install/run.uc)) применяет его на **commit-пути**, уже
после успешных шагов и health-check. Если `passwd` не сработал — честный warning «установите
пароль вручную по SSH», установка при этом успешна (пароль — вторичная защита, не data-plane).

## Граница доверия

Пароль приходит дважды проверенным: синхронно на ubus-границе (`install.root_password`,
`minlen` 8 — пользователь получает отказ сразу, до фоновой установки) и ещё раз в `apply.uc`
(stdin — тоже граница). Совпадение двух полей — забота UI; движок видит уже одно значение.

## Использование

```sh
echo '{"root_password":"correct horse battery"}' | ucode -R engine/steps/rootpass/apply.uc --dry-run
```

## Тесты

`make test-engine`. Покрыто: пустой/не строка, короче минимума (сообщение называет минимум),
валидные 8+ символов (включая фразу с пробелами).
